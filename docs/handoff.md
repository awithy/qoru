# qoru Handoff Notes

## Current Status

`qoru` has progressed from CLI/config scaffolding to a working QUIC/mTLS client-server path with a custom binary control protocol and server-side TCP proxying.

The current implemented flow is:

```text
qoru client
  -> QUIC/mTLS connect to qoru server
  -> open QUIC stream
  -> send binary ConnectTCPRequest(target)

qoru server
  -> accept QUIC/mTLS connection
  -> accept stream
  -> read ConnectTCPRequest(target)
  -> dial TCP target
  -> proxy QUIC stream <-> TCP target
```

The missing piece for a user-facing TCP proxy is the client-side local TCP listener.

## Implemented Pieces

### CLI

Commands:

```sh
qoru client
qoru server
qoru print-config
```

Global config flag:

```sh
-c, --config
```

Config default lookup order:

```text
./qoru.yaml
./qoru.yml
/etc/qoru/config.yaml
/etc/qoru/config.yml
```

The CLI uses function injection for command runners so tests can avoid starting real network services.

### Config

YAML config is implemented with validation for:

- common identity fields
- client mode fields
- server mode fields

Example configs live in:

```text
examples/config/client.yaml
examples/config/server.yaml
```

### Dev certs

Dev certificates are generated by:

```sh
./dev/gen-certs.sh
```

or:

```sh
make gen-dev-certs
```

Generated certs live under `dev/certs/` and are gitignored.

### TLS/mTLS

Implemented in `internal/identity`.

Current behavior:

- TLS 1.3 minimum
- private CA from config only; system CAs are not used
- server requires verified client certs
- client verifies server cert against `server.id`
- ALPN is `qoru/1`

### Protocol

Implemented in `internal/protocol`.

Frame envelope:

```text
version uint8
type    uint8
length  uint16 big endian
payload []byte
```

Current message:

```go
ConnectTCPRequest{Target string}
```

Payload:

```text
target_len uint16 big endian
target     []byte
```

After the initial `ConnectTCPRequest`, the stream currently carries raw TCP bytes.

### Server

Implemented in `internal/server`.

Current behavior:

- validates config
- loads server TLS config
- starts QUIC listener
- accepts connections
- accepts a stream
- reads `ConnectTCPRequest`
- dials TCP target
- proxies bytes both ways

Current limitation: server handling is still minimal and effectively handles one stream in the current connection handler path.

### Client

Implemented in `internal/client`.

Current behavior:

- validates config
- connects to server with QUIC/mTLS
- opens stream
- sends `ConnectTCPRequest`
- `ConnectTCP` helper returns the stream for tests

Current limitation: `qoru client` does not yet bind local `tcp_forwards.listen` ports.

## Recommended Next Step

Implement the client-side local TCP listener.

Desired behavior:

```text
local app
  -> tcp_forwards.listen
  -> qoru client
  -> QUIC stream
  -> qoru server
  -> tcp_forwards.target
```

For each configured `tcp_forwards` entry, `qoru client` should listen on `listen`. For each accepted local TCP connection:

1. connect to qoru server, or reuse an existing QUIC connection
2. open a new QUIC stream
3. send `ConnectTCPRequest{Target: forward.Target}`
4. proxy bytes both ways:

```text
local TCP conn <-> QUIC stream
```

## Suggested Small Implementation Plan

Keep the next step small:

1. Add client-side TCP listener for only the first `tcp_forwards` entry.
2. On accept, call existing `ConnectTCP`.
3. Add a client-side proxy helper similar to server `proxyTCP`.
4. Add an integration test:
   - start TCP echo target
   - start qoru server on `127.0.0.1:0`
   - start qoru client on `127.0.0.1:0`
   - connect to client local listener
   - write `ping`
   - expect `ping` back
5. Then generalize to all `tcp_forwards` entries.

## Design Choices to Revisit Soon

- Whether the client should keep one QUIC connection open and multiplex streams, or dial per local TCP connection initially.
- Whether server should support multiple streams per QUIC connection immediately.
- How to report target dial failures back to the client.
- Whether to add log-level/log-format flags.
- How to extract and use peer node IDs from client certificates.
- Server-side access policy for requested targets.
- Future encrypted data frames for multi-hop/end-to-end payload confidentiality.

## Known Caveats

- `qoru client` is not yet a long-running local proxy.
- Server target policy does not exist; authenticated clients can request any TCP target.
- Current stream protocol is one-hop oriented: control frame followed by raw TCP bytes.
- Future multi-hop/end-to-end encryption will likely require framed encrypted data payloads.
- Dev cert tests assume `dev/certs/` exists. Run `make gen-dev-certs` if cert-related tests fail.
